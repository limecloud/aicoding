# Token 预算与取舍策略

AI Coding 不是越全越好。完整 Harness、长上下文、审查 Agent 组、安全红队、端到端测试、链路记录和证据包，都会消耗 Token、时间、金钱和人类注意力。成熟团队不会每次都跑满流程，而是根据风险、预算和可逆性选择合适档位。

## 核心提醒

> Token 预算不充裕时，不要盲目追求“最全流程”。应保留关键事实源和高风险闸门，压缩低风险环节；必要时牺牲部分自动化效率，换成人工辅助、抽样审查或更小的任务切片。

这里的预算不只指模型 Token，还包括：

- **上下文预算**：能放多少规则、代码、日志、截图和历史决策。
- **推理预算**：能跑几轮规划、实现、审查和修复。
- **工具预算**：能跑多少测试、浏览器、CI、扫描和 MCP 工具。
- **人类注意力预算**：人能看多少摘要、风险和证据。
- **时间预算**：任务是否允许等待长流程完成。

## 三档执行模式

| 模式 | 适用场景 | 必须保留 | 可以压缩 |
| --- | --- | --- | --- |
| 简化模式 | 低风险、小改动、预算紧 | 小切片、相关文件、最小验证、1 个 AI 审查 Agent、人工抽样 | 全仓扫描、完整审查组、耗时端到端测试、完整证据包 |
| 标准模式 | 普通功能、可回滚变更 | 上下文包、自检、需求/代码/测试审查、关键路径端到端测试、简版证据 | 多轮红队、全链路观测、完整本地评估集 |
| 严控模式 | 权限、租户、支付、数据迁移、生产发布 | 完整上下文、审查组、测试质疑、安全红队、链路记录、回滚、人工关口 | 不建议压缩；只能缩小任务切片 |

## 风险与预算矩阵

| 风险 / 预算 | 预算紧 | 预算正常 | 预算充裕 |
| --- | --- | --- | --- |
| 低风险 | 简化模式：AI 实现 + 最小验证 + 人工抽样 | 标准精简版 | 标准模式 |
| 中风险 | 先缩小切片，再走标准精简版 | 标准模式 | 标准模式 + 定向红队审查 |
| 高风险 | 不压缩流程；先拆分或人工主导 | 严控模式 | 严控模式 + 本地评估集 + 发布演练 |

判断公式：

```text
所需严谨度 = 风险等级 * 影响范围 * 不可逆程度
如果所需严谨度超过预算：先缩小范围，不要先砍验证。
```

## 预算不足时优先保留什么

不要牺牲这些环节：

- **事实源**：仓库规则、契约、运行命令、风险边界必须明确。
- **任务切片**：预算越紧，任务越要小。
- **最小验证**：至少要有能证明主要行为的命令或手工步骤。
- **独立审查**：至少让另一个 AI 或人看高风险点，不能只相信实现 Agent。
- **回滚判断**：涉及生产、数据、权限时，必须知道如何撤回。

可以压缩这些环节：

- 把完整上下文改成仓库地图 + 相关文件 + 关键规则。
- 把完整审查 Agent 组改成 1-2 个定向审查 Agent。
- 把完整端到端测试改成关键路径冒烟验证 + 失败截图。
- 把长报告改成阻断级/高风险问题 + 缺失证据。
- 把多轮自动修复改成人先合并问题，再让 AI 做一次最小修复。

## 人工配合的替代手段

预算不足时，不要硬让 AI 独自完成全部工作。更稳的做法是把人放在“压缩上下文”和“裁决风险”的位置：

| 预算问题 | 人工配合方式 |
| --- | --- |
| 上下文放不下 | 人先指出 3-8 个关键文件和事实源 |
| AI 计划发散 | 人先写清非目标、边界和退出条件 |
| 审查太贵 | 人指定审查 Agent 只查安全、测试或兼容风险 |
| 端到端测试太慢 | 人提供手工验收脚本，AI 负责记录证据 |
| 问题太多 | 汇总裁决 Agent只输出阻断级、高风险和需要人工裁决项 |
| 多轮修复消耗大 | 人先合并同类问题，再让 AI 做一次最小修复 |

## 提示词模板：预算受限模式

```text
你现在处于 Token 预算受限模式。不要追求全量审查。

目标：完成这个小切片，并给出最小可信证据。
预算策略：
- 只读取与任务直接相关的文件；如需更多上下文，先说明原因。
- 不做无关重构，不扩展范围。
- 验证优先级：类型检查/单测 > 关键路径冒烟验证 > 可选端到端测试。
- 审查只输出阻断级和高风险，不输出风格建议。
- 如果风险超过预算，停止并建议缩小任务或转人工裁决。

输出：
1. 修改摘要
2. 已使用上下文
3. 已运行验证
4. 阻断级/高风险问题
5. 需要人工裁决的问题
```

## 提示词模板：高风险不可压缩

```text
这是高风险变更。不要因为 Token 或时间不足而跳过验收。

如果预算不足，请优先做以下动作：
1. 缩小任务切片。
2. 明确必须保留的安全、数据、发布检查。
3. 标出无法完成的证据。
4. 请求人类裁决：延期、拆分，还是人工执行关键步骤。

禁止：
- 用“测试通过”替代安全审查。
- 用“看起来没问题”替代证据。
- 在缺少回滚路径时建议发布。
```

## 最终原则

AI 原生的目标不是“让 AI 做最多事情”，而是“用最少必要成本获得足够可信的交付结论”。预算紧张时，正确答案通常不是塞更多上下文，而是缩小任务、保留关键闸门、让人补足判断。
