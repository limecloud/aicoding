# AI Coding 落地路线图

不要从“全自动软件工程师”开始。AI Coding 的落地应该按风险、成熟度和 Token 预算渐进推进。流程也不是越全越好：预算不足时先缩小任务，再选择必要防线。

## 90 天路线图

### 0-30 天：建立事实源和小切片

- 建立 `AGENTS.md` / `CLAUDE.md`。
- 统一验证命令。
- 任务强制写非目标和验收标准。
- AI 只做小切片实现。
- 人类仍做最终 review。
- 建立简化 / 标准 / 严控 三档预算策略。

交付物：

```text
AGENTS.md
上下文包模板
验证契约模板
需求切片模板
```

### 31-60 天：引入 AI 先审

- 引入独立 AI 审查 Agent。
- 引入测试质疑 Agent。
- 高风险任务引入安全红队 Agent。
- 让汇总裁决 Agent 汇总问题清单。
- PR 上只展示阻断级/高风险。
- 按风险触发审查 Agent，不对所有小改动跑满审查组。

交付物：

```text
review-spec.md
review-tests.md
review-security.md
judge.schema.json
ai-acceptance-report.md
```

### 61-90 天：建立 Harness 和指标

- 接入 CI / GitHub Action。
- 保存证据产物。
- 建立人工关口。
- 建立本地评估集。
- 追踪审查命中率、逃逸缺陷和返工次数。
- 追踪 Token、CI 时间和人类审查时间，定期删除低价值检查。

交付物：

```text
make ai-review-changed
CI 产物
local-evaluation-suite
team-metrics-dashboard
risk-waiver-log
budget-policy
```

## 预算档位落地

| 档位 | 触发条件 | 必须做 | 不建议做 |
| --- | --- | --- | --- |
| 简化模式 | 低风险、可回滚、小范围代码差异 | 最小上下文、最小验证、1 个审查 Agent、人类抽样 | 完整审查 Agent 组、耗时端到端测试、完整证据包 |
| 标准模式 | 普通功能、影响单一模块 | 上下文包、自检、需求/代码/测试审查 Agent、关键路径证据 | 默认打开安全红队 |
| 严控模式 | 认证、租户、支付、数据、生产发布 | 完整审查 Agent 组、安全红队、回滚、人工关口 | 以预算不足为由跳过关键验收 |

如果预算不够支撑目标档位，优先拆任务或让人类补充上下文，不要把高风险任务降级成低风险流程。

## 成熟度门槛

| 能力 | 没有时不要做什么 |
| --- | --- |
| 小切片 | 不要让 AI 做大功能 |
| 验证命令 | 不要让 AI 自动修改关键代码 |
| AI 审查 Agent | 不要把 AI 生成代码直接交给人全量审 |
| 安全红队 Agent | 不要让 AI 修改认证、租户、支付 |
| 验收证据包 | 不要发布高风险变更 |
| 人工关口 | 不要自动合并或自动发布 |

## 团队规则

- 低风险自动化，高风险强治理。
- AI 生成的测试必须被另一个 AI 审查。
- 需求说明、代码、测试、证据必须能互相映射。
- 任何风险豁免必须有负责人和过期时间。
- 每周复盘 AI 错误，更新规则而不是口头抱怨。
