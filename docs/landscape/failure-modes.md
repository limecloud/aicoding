# AI Coding 失败模式

AI Coding 的价值来自规模化执行，但失败也会被规模化。团队需要把常见失败模式前置到 Harness 引擎、审查 Agent 和人工关口中。

## 常见失败模式

| 失败模式 | 表现 | 防线 |
| --- | --- | --- |
| 上下文漂移 | AI 修错路径、复活旧实现 | 仓库地图、事实源、current/compat 标记 |
| 需求膨胀 | 顺手做无关重构或新功能 | 非目标、小切片、代码差异范围审查 |
| 弱测试 | 测试只覆盖正常路径，或 mock 掉真实边界 | 测试质疑 Agent、变异测试思路、负例 |
| 自我验收 | 实现 Agent 解释自己为什么对 | 独立审查 Agent、只读审查 |
| 安全盲区 | 认证 / 租户 / 数据 被顺带改坏 | 安全红队 Agent、risk trigger |
| 工具误用 | 跑错命令、写生产、泄漏 密钥 | 工具白名单、沙箱、人工关口 |
| 证据缺失 | 报告说通过，但无日志/截图/链路记录 | 验收证据包 |
| 过度自动化 | 低价值检查消耗大量 Token | 预算档位、定期删减 Harness |
| 人的瓶颈 | 人被迫读全部代码差异 | 汇总裁决 Agent 摘要、阻断级/高风险聚合 |
| 过度信任 | 合并 AI 结论但没有裁决负责人 | 风险豁免、负责人、过期时间 |

## 失败复盘问题

```text
1. 哪个事实源没有进入上下文？
2. 哪个审查 Agent 本应发现但没发现？
3. 测试为什么没有证明真实需求？
4. 是否因为 Token/时间预算压缩了不该压缩的环节？
5. 下次应该更新规则、工具、模板还是人工关口？
```

## 防线优先级

预算不足时，按这个顺序保留防线：

1. 小切片和非目标。
2. 事实源和相关文件。
3. 最小验证命令。
4. 独立 AI 审查。
5. 高风险人工门。
6. 完整证据包和长期评估集。

## 让失败进入知识库

每次逃逸缺陷都应沉淀为：

- 新的仓库规则。
- 新的审查 Agent 检查项。
- 新的回归测试或评估样本。
- 新的风险触发器。
- 删除一个低价值检查，避免流程膨胀。
