# 需求与任务切片

AI 可以很快实现模糊需求，但模糊需求会把复杂度转移到验收阶段。AI Coding 中，需求说明 的核心价值是建立“可机器审查”的验收边界。

## 需求说明 最小字段

| 字段 | 作用 |
| --- | --- |
| 用户目标 | 说明为什么要做，避免技术实现跑偏 |
| 当前问题 | 说明没有这个功能时的痛点 |
| 范围内 | 明确本次必须实现的行为 |
| 范围外 | 防止 AI 顺手实现未来功能 |
| 验收标准 | 每条都应映射到测试、链路记录、截图或人工裁决 |
| 风险面 | 权限、数据、并发、发布、兼容、成本 |
| 回滚方式 | 说明失败后如何关闭或撤销 |

## 切片规则

好的 AI 任务切片应该是垂直的，而不是按技术层横切。

不推荐：

```text
PR 1: 只改数据库
PR 2: 只改后端 API
PR 3: 只改前端页面
```

推荐：

```text
PR 1: 用户可以创建一个最小资源，并在列表看到它
PR 2: 用户可以编辑该资源，并看到错误提示
PR 3: 管理员可以审核该资源，并记录审计日志
```

## AI 切片提示词

```text
请不要写代码。请把这个 需求说明 拆成 3-7 个可独立验收的垂直切片。

每个切片必须包含：
1. 用户可见结果。
2. 需要触达的入口层、服务层、存储层、旁路层。
3. 自动化测试建议。
4. AI 审查 Agent需要重点审查的风险。
5. 明确不做的内容。
6. 回滚或关闭方式。

如果某个切片预计超过 30 分钟人工审查，请继续拆小。
```
