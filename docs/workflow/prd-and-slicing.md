# 需求、PRD 与任务切片

AI Coding 之前，最容易被低估的是 PRD。很多人以为“需求写清楚一点”就够了，但真正进入 AI Coding 后，PRD 还要承担另一层职责：把业务目标、产品行为、工程边界、验收证据和发布风险，整理成 Agent 可以执行、审查和复盘的输入协议。

这里的 PRD 不是传统大文档，也不是只给产品经理看的说明书。它是全栈、全业务周期的交付契约：从机会判断、方案设计、研发实现、验收发布，到上线后的观测、反馈和迭代，都应该能从 PRD 追溯回来。

## AI Coding 前为什么更需要 PRD

AI 写代码很快，但它不会天然知道：

- 这个需求解决的是谁的真实问题。
- 哪些用户路径必须优先保证。
- 哪些行为是非目标，不能顺手做。
- 哪些规则来自业务、运营或客服，不在代码里。
- 什么证据可以证明“做对了”。
- 上线后看什么指标，失败后怎么回滚。

如果 PRD 只写“做一个 xx 功能”，AI 会把缺失信息补成自己的假设。最后代码看起来完整，但业务闭环不一定成立。

## PRD 要覆盖全业务周期

| 周期 | PRD 要说明什么 | AI Coding 怎么消费 |
| --- | --- | --- |
| 机会判断 | 用户是谁、痛点是什么、为什么现在做 | 防止 Agent 做成技术演示 |
| 业务目标 | 要提升什么指标、降低什么成本、支持什么流程 | 让验收不只看页面是否存在 |
| 用户路径 | 正常路径、异常路径、边界路径、权限差异 | 生成端到端测试和视觉验收脚本 |
| 规则与约束 | 价格、权限、状态机、审批、通知、数据口径 | 约束 service、schema、API、UI 状态 |
| 工程边界 | 范围内、范围外、兼容策略、事实源 | 防止 AI 顺手重构或造第二套实现 |
| 数据与运营 | 埋点、报表、后台处理、客服可见性 | 让发布后能观测和运营 |
| 风险与发布 | 灰度、回滚、监控、迁移、安全风险 | 触发安全红队、发布审查和人工关口 |
| 复盘迭代 | 指标回看、用户反馈、后续版本 | 把线上反馈回流到下一轮 PRD |

## PRD 不要写成什么样

| 写法 | 问题 |
| --- | --- |
| “做一个订单管理页面” | 没有角色、状态、权限、异常路径 |
| “参考竞品 A” | Agent 不知道哪些细节要抄，哪些不能抄 |
| “后台加个导出” | 没有数据范围、权限、审计、性能要求 |
| “优化体验” | 没有目标路径、成功标准和截图证据 |
| “先做 MVP” | 如果不写非目标，AI 会自己脑补完整功能 |

## 好 PRD 的判断标准

一份适合 AI Coding 的 PRD，至少应该能回答：

1. 这个需求解决哪个业务问题？
2. 谁使用？谁审核？谁运营？谁承担风险？
3. 正常路径和失败路径分别是什么？
4. 哪些行为必须自动化验证？
5. 哪些判断必须由人裁决？
6. 哪些代码事实源不能被 AI 顺手改掉？
7. 上线后用什么指标判断成功？
8. 出问题怎么灰度、关闭或回滚？

如果这些问题回答不了，先不要让 AI 大规模写代码。应该先让 AI 帮你访谈、补问题、拆切片。

## 用 AI 反向追问 PRD

PRD 不完整时，不要直接进入编码。先让 AI 扮演产品、工程、安全、运营四个角色追问。

```text
你不是实现 Agent。请审查这份 PRD 是否足以进入 AI Coding。

从四个视角追问：
1. 产品：用户目标、路径、状态、边界是否清楚？
2. 工程：调用链、事实源、数据模型、兼容策略是否清楚？
3. 安全：权限、租户、数据、审计、外部输入是否清楚？
4. 运营：上线后如何配置、监控、客服处理和回滚？

输出：
- 必须补充的问题。
- 可以后置的问题。
- 不影响当前切片的问题。
- 是否允许进入任务切片。
```

## 从 PRD 到任务切片

好的 AI 任务切片应该是垂直的，而不是按技术层横切。

不推荐：

```text
PR 1: 只改数据库
PR 2: 只改后端 API
PR 3: 只改前端页面
```

推荐：

```text
PR 1: 用户可以创建一个最小资源，并在列表看到它
PR 2: 用户可以编辑该资源，并看到错误提示
PR 3: 管理员可以审核该资源，并记录审计日志
```

每个切片必须能独立回答：

- 用户可见结果是什么？
- 涉及哪些层：入口、服务、存储、UI、契约、发布？
- 需要哪些测试和运行证据？
- 审查 Agent 要重点看什么风险？
- 哪些内容明确不做？
- 如何回滚或关闭？

## AI 切片提示词

```text
请不要写代码。请把这份 PRD 拆成 3-7 个可独立验收的垂直切片。

每个切片必须包含：
1. 用户可见结果。
2. 业务规则和用户路径。
3. 需要触达的入口层、服务层、存储层、UI、契约、旁路层。
4. 自动化测试建议。
5. AI 审查 Agent 需要重点审查的风险。
6. 明确不做的内容。
7. 回滚或关闭方式。

如果某个切片预计需要超过 30 分钟人工审查，请继续拆小。
```

## PRD 到验收证据的映射

不要让验收只停留在“测试通过”。PRD 中每条验收标准都应该能映射到证据。

| PRD 验收项 | 代码证据 | 测试证据 | 运行证据 | 人工裁决 |
| --- | --- | --- | --- | --- |
| 用户能创建资源 | API、service、UI 表单 | 单测、集成、端到端 | 截图、链路记录 | 否 |
| 无权限用户不能创建 | 权限中间件、service 校验 | 403 负例 | API 请求记录 | 否 |
| 错误提示符合品牌语气 | UI 文案、错误态 | 端到端或截图 | 设计验收截图 | 是 |
| 大客户灰度开启 | feature flag、租户白名单 | 配置测试 | 发布记录、监控 | 是 |

## 三档 PRD 详细度

不是所有需求都要写大 PRD。PRD 也要按风险和预算分档。

| 档位 | 适用场景 | 必须写清 |
| --- | --- | --- |
| 简化 PRD | 文案、小样式、低风险配置 | 目标、范围、验收、验证命令 |
| 标准 PRD | 普通功能切片 | 用户路径、业务规则、数据/API、测试、风险、发布 |
| 严控 PRD | 权限、租户、支付、数据迁移、生产发布 | 完整业务周期、状态机、安全、灰度、回滚、运营、人工关口 |

预算不足时，优先缩小 PRD 覆盖的切片，而不是省略高风险字段。
