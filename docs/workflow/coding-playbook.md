# 写代码实战方法

AI 写代码不能只靠一句“帮我实现”。更稳定的方式，是把编码拆成一条短闭环：**给验证信号 -> 先探索 -> 再计划 -> 小步实现 -> 自检修复 -> 独立审查**。

本页参考 [Claude Code Best Practices](https://code.claude.com/docs/en/best-practices) 的方法，但按本站的 AI Coding / Harness 体系重新整理，重点放在真实仓库里怎么让 Agent 写出可验收、可回滚、可维护的代码。

## 一句话原则

> 不要让 Agent 靠“看起来写完了”停下来。每次写代码前，都要先告诉它：做什么、不做什么、怎么验证、什么时候停、哪些风险必须交给人。

## 编码前：先给可运行的验收信号

如果没有测试、构建、截图、脚本或人工验收脚本，Agent 只能靠主观判断“应该差不多了”。这时人会重新变成验证循环。

| 任务类型 | 推荐给 Agent 的验收信号 |
| --- | --- |
| 纯函数 / 工具函数 | 单测、边界用例、输入输出样例 |
| API / 服务逻辑 | 单测 + 集成测试 + 关键错误码 |
| UI 改动 | 截图、设计稿、Playwright 链路、响应式检查 |
| Bug 修复 | 先写失败用例，再修复，最后证明用例通过 |
| 重构 | 行为不变测试、公开接口不变、关键路径回归 |
| 数据迁移 | dry-run、行数对账、可重复执行、回滚方案 |

最小提示词：

```text
请实现这个切片，但不要只说完成。

完成前必须提供：
1. 修改了哪些文件。
2. 运行了哪些验证命令，结果是什么。
3. 哪些需求条目对应到哪些代码和测试。
4. 没有验证的部分，以及原因。
5. 剩余风险和需要人工裁决的问题。
```

## 标准编码流程

### 1. Explore：先读，不写

让 Agent 先理解事实源，而不是直接生成代码。

适合让它读：

- 仓库规则：`AGENTS.md`、`CLAUDE.md`、Copilot instructions、项目 README。
- 相关入口：页面、路由、API handler、command、job。
- 业务层：service、domain、use case、权限判断。
- 存储层：repository、migration、schema、query。
- 现有测试：相似用例、fixture、mock 方式。
- 契约：OpenAPI、SDK、types、事件协议。

探索提示词：

```text
先不要写代码。
请阅读相关文件，回答：
1. 当前能力的唯一事实源在哪里？
2. 这次改动应该走哪条调用链？
3. 是否存在 current / compat / deprecated / dead 路径？
4. 现有测试模式是什么？
5. 哪些文件最可能需要修改？
6. 哪些地方不能碰？

只输出结论和文件路径，不要实现。
```

### 2. Plan：写计划，让人能改

计划不是写长篇方案，而是把风险提前暴露出来。好的计划应该短、可编辑、能映射到验证。

计划必须包含：

- 本切片目标和非目标。
- 预期修改文件。
- 数据流和调用链变化。
- 测试计划：正例、负例、边界、回归。
- 风险点：权限、数据、兼容、性能、发布。
- 验证命令。
- 回滚或关闭方式。

计划提示词：

```text
请基于刚才的探索结果写一个最小实现计划。

要求：
- 只覆盖当前切片，不扩大范围。
- 明确每个文件为什么要改。
- 明确哪些验证必须跑。
- 标出需要人工裁决的风险。
- 如果你认为任务太大，请继续拆小，而不是直接实现。
```

什么时候可以跳过计划：

- 改文案、改拼写、单行配置。
- 明确的小 bug，能一句话描述 diff。
- 只补一个非常明确的测试。

什么时候必须计划：

- 多文件修改。
- 不熟悉的模块。
- 涉及权限、租户、支付、数据、发布、CI/CD。
- 需要改变 API、SDK、类型、迁移或兼容策略。

### 3. Code：小步实现，不要一口气写完

进入实现后，要求 Agent 按小步提交思路工作，但不一定真的提交 git：

1. 先补或定位验证用例。
2. 实现最小行为。
3. 跑最小验证。
4. 修明确失败。
5. 再补边界和负例。
6. 最后跑完整或变更面验证。

实现提示词：

```text
请按计划实现，但遵守这些限制：
1. 只改计划内文件；如需新增文件，先说明原因。
2. 不引入新依赖，除非明确说明为什么必须引入。
3. 不删除或弱化旧测试。
4. 不把重构和行为变化混在一起。
5. 不新增长期 compat；如果必须加兼容层，写清退出条件。
6. 每完成一个小步，运行最小验证并根据失败修复根因。
```

### 4. Verify：修根因，不修表象

Agent 很容易为了让测试通过而“压住错误”：跳过断言、扩大 mock、吞异常、改快照、删测试。这些都必须禁止。

验证提示词：

```text
现在运行验证。
如果失败：
- 先解释失败根因。
- 不要删除测试、弱化断言、吞掉错误或只改快照。
- 修复后重新运行同一条验证命令。
- 如果无法判断根因，停止并输出 needs_human。
```

### 5. Self-check：实现者只做自检，不做最终验收

实现 Agent 可以自检，但不能批准自己。

自检报告模板：

```markdown
# 实现自检报告

## 目标
- 

## 实际改动
- 

## 需求到证据映射
| 需求 | 代码 | 测试 / 运行证据 | 状态 |
| --- | --- | --- | --- |

## 已运行验证
- 命令：
- 结果：

## 未验证内容
- 原因：

## 风险
- 

## 建议审查重点
- 
```

## Prompt 怎么写才像工程任务

### 差提示词

```text
帮我把登录优化一下。
```

问题：目标不清、边界不清、验证不清，Agent 只能猜。

### 好提示词

```text
目标：修复用户 session 过期后刷新页面会进入空白页的问题。

范围内：
- 只处理 session 过期后的前端状态恢复。
- 用户应看到登录页或明确错误提示。

范围外：
- 不改登录协议。
- 不改后端 token 策略。
- 不重构全局 auth store。

请先读：
- apps/web/src/auth/*
- apps/web/src/routes/*
- 现有 auth 相关测试

实现要求：
1. 先写或补一个能复现空白页的失败用例。
2. 再实现最小修复。
3. 运行相关测试。
4. 输出需求、代码、测试、风险映射。
```

## 上下文要具体，不要堆满

给 Agent 的上下文越多，不一定越好。关键是给它**和当前切片直接相关**的信息。

| 应该给 | 不应该给 |
| --- | --- |
| 相关文件路径 | 整个仓库 dump |
| 错误日志和复现步骤 | “你自己看着办” |
| 现有实现模式 | 长篇教程和无关背景 |
| 验证命令 | 几百行无裁剪日志 |
| 明确非目标 | 希望它顺便优化所有问题 |
| 风险面 | 让它自行决定哪些安全规则可忽略 |

## 仓库规则文件怎么写

无论是 `AGENTS.md`、`CLAUDE.md` 还是 Copilot instructions，都应该短而硬。它们不是项目百科，而是高频约束。

应该写：

- 默认验证命令。
- 代码风格里真正不同于默认习惯的部分。
- 架构事实源和禁止扩展的路径。
- 分层边界：入口层、服务层、存储层、旁路层。
- 危险操作规则：删除、迁移、生产 API、依赖、发布。
- 常见坑：本地环境、测试数据、租户、权限、时区。

不应该写：

- Agent 读代码就能知道的东西。
- 大段 API 文档，链接即可。
- 文件逐个介绍。
- 长篇教程。
- “写干净代码”这种没有执行力的空话。

规则文件太长时，Agent 会忽略真正重要的规则。更好的做法是：公共规则放主文件，领域知识放技能、子 Agent 或按需上下文包。

## 会话管理：不要把一个会话用到天荒地老

长会话会累积无关信息、失败尝试和旧决策，后面越写越容易漂。

建议：

- 不相关任务之间清空上下文。
- 同一问题纠正超过两次，停止继续补丁式纠错，重新开干净上下文。
- 长任务压缩上下文时，明确保留：修改文件、关键决策、验证命令、剩余风险。
- 大功能先让 Agent 访谈或写规格，再开新会话按规格实现。
- 多方向探索用多个会话或 worktree，避免互相污染。

## 子 Agent 怎么用

子 Agent 的价值不是“多几个 AI 热闹一点”，而是隔离上下文。

适合子 Agent 的任务：

- 调查认证、权限、数据流等需要读很多文件的问题。
- 在实现后做独立代码审查。
- 安全红队。
- 测试质疑。
- 对比两个实现方案。

不适合子 Agent 的任务：

- 多个 Agent 同时改同一批文件。
- 没有统一裁决的并行实现。
- 让子 Agent 带着实现者解释去审查。

子 Agent 审查提示词：

```text
你是独立审查 Agent。不要修改代码。

输入只有：计划、当前 diff、测试结果、仓库规则。
请只找会影响正确性、需求覆盖、安全、数据、兼容和发布的缺口。
不要输出风格偏好。
每个问题必须给文件位置、证据、影响和建议修复方式。
```

## 钩子、权限和工具

真正稳定的 AI Coding 不能只靠模型自觉。能自动化的硬规则，应尽量放到工具层：

| 能力 | 作用 |
| --- | --- |
| 权限白名单 | 允许常规测试、lint、构建，拦住生产 API、迁移、删除 |
| 沙箱 | 限制文件系统、网络和系统资源访问 |
| CLI 工具 | 用 `gh`、云厂商 CLI、监控 CLI 获取真实状态，减少复制粘贴 |
| MCP | 连接 issue、文档、Figma、数据库只读查询、监控等外部事实源 |
| Hooks | 每次编辑后自动跑格式化、类型检查，或阻止写入危险目录 |
| Skills | 把领域规则和高频工作流按需加载，避免主上下文臃肿 |

## 不同任务的写法

### 小 bug

```text
请修复这个 bug。先写一个最小失败用例复现，再做最小修复。
不要扩大范围。修复后运行相关测试，并说明为什么这是根因。
```

### 新功能

```text
请先探索并计划，不要直接写代码。
计划必须包含：用户路径、涉及文件、数据流、测试、风险、回滚。
我确认计划后，再按小切片实现。
```

### UI 改动

```text
请根据截图实现 UI 调整。
实现后用浏览器打开页面，截图对比：布局、间距、颜色、响应式、空状态、错误态。
列出差异并修复明显问题。
```

### 重构

```text
这是行为不变重构。
不得改变公开 API、用户行为和测试语义。
先列出现有行为保护网；如果保护网不足，先补测试。
每一步重构后运行相关验证。
```

### 大规模迁移

```text
请先选 2-3 个代表性文件试迁移。
总结失败模式和提示词修正后，再生成批量迁移计划。
批量执行必须限制 allowed tools，并输出 OK / FAIL 列表。
```

## 常见失败模式

| 失败模式 | 表现 | 处理方式 |
| --- | --- | --- |
| 直接开写 | 写了很多，但方向错 | 先探索和计划 |
| 上下文太满 | 后半段开始忘规则 | 清空、压缩、拆会话 |
| 规则文件太长 | 关键规则被忽略 | 删减规则，改用 hook / skill |
| 为通过测试而改测试 | 测试绿了但风险更大 | 禁止弱化测试，要求解释根因 |
| 无限调查 | 读很多文件但不收敛 | 限定调查问题，或交给子 Agent |
| 自我验收 | 实现者说自己没问题 | 新上下文审查 Agent 复核 |
| 过度修复 | 为一个问题引入大重构 | 只修裁决报告里的阻断级和高风险 |

## 三档执行强度

| 档位 | 适用场景 | 写代码方式 |
| --- | --- | --- |
| 简化模式 | 文案、小样式、低风险配置 | 可跳过完整计划，但必须有最小验证和 diff 自检 |
| 标准模式 | 普通功能切片 | Explore -> Plan -> Code -> Verify -> Self-check -> AI 审查 |
| 严控模式 | 权限、租户、支付、数据迁移、生产发布 | 标准模式 + 安全红队 + 测试质疑 + 发布证据 + 人工关口 |

预算不足时，不要让高风险任务降级。正确做法是缩小切片，或让人补充关键上下文和手工验收证据。
