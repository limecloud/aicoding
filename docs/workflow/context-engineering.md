# 上下文工程

AI Coding 的质量上限，很大程度取决于上下文。上下文工程不是把所有文件都塞给模型，而是选择最小充分事实：既让 AI 能做正确决策，又避免被无关内容带偏。

## 上下文分层

| 类型 | 例子 | 使用方式 |
| --- | --- | --- |
| 规则上下文 | AGENTS、代码风格、禁区 | 每次任务默认加载 |
| 任务上下文 | 需求说明、issue、验收标准 | 每个切片加载 |
| 代码上下文 | 相关入口、服务、测试、契约 | 按调用链选择 |
| 历史上下文 | ADR、相关 PR、事故复盘 | 只在决策相关时加载 |
| 运行上下文 | 测试命令、日志、链路记录、CI 输出 | 审查和修复阶段加载 |

## 好上下文的标准

- 能说清唯一事实源在哪里。
- 能说清当前任务不应该碰哪里。
- 能说明验证命令和失败解释方式。
- 能展示已有模式，而不是让 AI 发明新模式。
- 足够小，让审查 Agent 能独立判断。

## 上下文包模板

```markdown
# 上下文包

## 任务
- 需求说明 / issue:
- 本切片目标:
- 非目标:

## 仓库规则
- 必读规则文件:
- 禁止事项:
- 默认验证命令:

## 相关代码
- 入口:
- 服务:
- 存储:
- UI:
- 测试:
- 契约:

## 风险面
- 权限:
- 数据:
- 兼容:
- 发布:

## 验收证据
- 必须生成:
- 可选生成:
```
