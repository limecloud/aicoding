# 测试审查：Test Skeptic

AI 写单元测试和 E2E 还不够。最危险的情况是：实现错了，测试也按错误实现写，最后全部通过。Test Skeptic 的职责不是确认测试通过，而是攻击测试的可信度。

## 审查目标

Test Skeptic 必须回答：

- 测试是否覆盖 PRD 的每条验收标准？
- 是否只有 happy path？
- 是否验证了真实业务结果，而不是只验证函数被调用？
- 是否 mock 掉了真正应该集成验证的边界？
- 如果关键代码被故意改坏，测试会不会失败？
- 是否有实现和测试同时错误但仍通过的场景？

## 常见弱测试信号

| 信号 | 风险 |
| --- | --- |
| 只断言 `toBeTruthy` | 不证明业务正确 |
| 只验证 UI 元素存在 | 不证明数据写入或权限正确 |
| mock 了整个 service | 无法发现集成边界错误 |
| 没有负例 | 权限、非法输入、重复提交容易漏 |
| 测试名称复述实现 | 没有表达业务行为 |
| 删除旧测试但无说明 | 可能掩盖回归 |

## Mutation 场景

Test Skeptic 应提出 mutation 场景，而不是只说“建议增加测试”。

```text
1. 删除 tenantId 查询条件，现有测试是否失败？
2. 把权限判断从 AND 改成 OR，现有测试是否失败？
3. API 返回 500，前端 E2E 是否验证错误状态？
4. 重复点击提交按钮，是否产生重复记录？
5. 空数组、null、超长字符串是否被覆盖？
```

## Prompt

```text
你是 Test Skeptic。不要修改代码。

请审查当前 diff 中新增或修改的测试，目标是证明这些测试可能无效。

必须输出：
1. PRD 验收标准到测试的覆盖矩阵。
2. 弱断言、mock 过度、happy path-only 的测试。
3. 至少 5 个 mutation 场景，并判断现有测试是否会失败。
4. 必须补的负例测试。
5. 如果测试不足以证明需求正确，verdict 必须是 needs_human 或 block。
```
