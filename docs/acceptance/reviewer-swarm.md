# 审查 Agent 组

审查 Agent 组不是为了模拟一个“全能资深审查人”，而是让多个独立角色从不同角度审同一个变更。它的价值在于分工、隔离和证据化：实现者不审自己，测试问题由测试质疑者看，安全问题由安全红队看，最终再由汇总裁决 Agent去重和定级。

## 推荐角色

| 角色 | 输入 | 输出 | 重点 |
| --- | --- | --- | --- |
| 需求审查 Agent | 需求说明、验收标准、代码差异 | 需求覆盖矩阵 | 漏实现、需求漂移、夹带非目标 |
| 代码审查 Agent | 代码差异、相关代码 | 缺陷清单 | 边界、错误处理、重复、复杂度 |
| 测试质疑 Agent | 测试、被测代码 | 测试缺口 | 弱断言、过度 mock、缺少负例 |
| 安全红队 Agent | 代码差异、权限、数据、配置 | 攻击路径 | 租户、权限、注入、密钥、CI/CD |
| 治理审查 Agent | 仓库规则、代码差异 | 规则违规 | current/compat、契约、发布事实源 |
| 体验审查 Agent | 页面、截图、链路记录 | 体验风险 | 空状态、错误态、响应式、可访问性 |
| 发布审查 Agent | 配置、迁移、运行手册 | 发布风险 | 灰度、监控、回滚、迁移 |

## 隔离原则

- 审查 Agent 只读取需求说明、仓库规则、代码差异和验证结果，不读取实现者的主观解释。
- 审查 Agent默认只读，不直接修改代码。
- 每个问题都要给出文件位置、证据和复现方式；没有证据就写入 `missing_evidence`。
- 不允许只输出“看起来没问题”。
- 不确定时输出 `needs_human`，交给人类负责人裁决。

## 输出示例

```json
{
  "reviewer": "security_red_team",
  "verdict": "block",
  "confidence": 0.82,
  "findings": [
    {
      "severity": "high",
      "type": "security",
      "file": "path/to/file.ts",
      "line": 123,
      "claim": "这里可能绕过租户隔离",
      "evidence": "查询条件只包含 userId，没有包含 tenantId",
      "repro": "使用 tenant A 的 Token 请求 tenant B 的资源",
      "suggested_fix": "在 service 层加入 tenantId 约束，并补充跨租户失败用例"
    }
  ],
  "missing_evidence": [],
  "human_decision_required": []
}
```

## 汇总裁决 Agent

汇总裁决 Agent只做五件事：

- 去重相同问题。
- 合并相关风险。
- 统一严重级别。
- 标出审查 Agent之间的冲突。
- 生成“人类只需要看的内容”。

输出要短，优先呈现阻断级和高风险问题。中低风险建议可以折叠到附录，避免重新把人拖回全量审查。
