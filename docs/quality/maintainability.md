# 可维护性

AI 很容易生成“当下能跑，但未来难维护”的代码。可维护性审查要重点拦住中心文件膨胀、一次性抽象、重复分叉和隐性兼容层。

## 常见问题

- 复制已有逻辑但改了细节，形成并行实现。
- 过早抽象一个只被调用一次的 helper。
- 为未来需求预留复杂配置。
- 把新行为塞进已经过大的中心文件。
- 生成未使用类型、未使用组件、未使用参数。
- 用注释解释复杂代码，而不是先把代码变简单。

## AI 可维护性规则

```text
1. 如果只调用一次，优先内联。
2. 如果重复模式尚未稳定，不要抽象。
3. 如果文件接近复杂度上限，新功能优先拆模块。
4. 如果存在 current 实现，不要创建第二套入口。
5. 如果新增 compat，必须写清退出条件和删除窗口。
6. 如果删除 dead code，要确认它确实没有运行时价值。
```

## 审查输出

可维护性问题项应关注行为成本，而不是审美：

- 这个重复会不会导致未来修缺陷时改一处漏一处？
- 这个抽象是否让代码更难读？
- 这个兼容层是否会变成长期事实源？
- 这个模块是否违反单一职责？
