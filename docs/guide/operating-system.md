# AI Coding 操作系统

AI Coding 操作系统不是一组零散提示词，而是一套稳定约束。它要回答几个最基本的问题：AI 从哪里拿事实，任务如何拆，代码如何写，谁来审，证据放哪里，人什么时候介入。

## 六层结构

```mermaid
graph TD
  A[目标与需求] --> B[上下文层]
  B --> C[实现层]
  C --> D[AI 审查层]
  D --> E[证据层]
  E --> F[人工裁决层]
  F --> G[发布与反馈]
  G --> A
```

| 层级 | 责任 | 典型产物 |
| --- | --- | --- |
| 目标与需求 | 说清做什么、不做什么、怎么验收 | 需求说明、issue、用户路径、非目标 |
| 上下文层 | 给 AI 最小充分事实 | AGENTS、架构图、契约、相关代码、运行命令 |
| 实现层 | 完成一个小切片 | 代码、测试、迁移、文档 |
| AI 审查层 | 多角色独立审查 | 需求审查、测试质疑、安全红队、治理审查 |
| 证据层 | 聚合可验证材料 | 测试结果、链路记录、截图、日志、风险表 |
| 人工裁决层 | 决定接受、退回还是豁免 | 阻断级决策、风险豁免、发布批准 |

## 默认交付单元

AI Coding 的交付单元不应该是“一个大需求”，而应该是“一个可运行、可审查、可回滚的垂直切片”。每个切片都应满足：

- 有单一用户价值或单一技术目标。
- 能说清入口、服务、存储、旁路影响面。
- 有自动化验证和 AI 审查结果。
- 有给人看的剩余风险摘要。
- 可以独立回滚或关闭。

## 最小事实源

每个仓库至少要准备这些事实源：

- `AGENTS.md`：仓库规则、禁区、默认验证命令。
- 架构说明：入口、服务、存储、事件、发布路径。
- API 契约：OpenAPI、SDK、类型、兼容策略。
- 测试入口：快速验证、完整验证、变更面检测。
- 发布运行手册：灰度、监控、回滚、事故处理。

没有事实源，AI 只能从局部代码猜全局规则。猜对了是运气，猜错了就是隐性风险。
