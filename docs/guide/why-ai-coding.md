# 为什么需要 AI Coding 体系

AI 已经显著降低“写代码”的成本，但软件交付的瓶颈会转移到需求澄清、上下文组织、审查、验收、发布和治理。如果团队只把 AI 当成补全工具，常见结果是：代码产出更快了，但审查、测试、返工和回滚也变多了。

AI Coding 体系的目标不是让 AI 替代工程师，而是重新设计工程流：让 AI 承担可规模化的实现、审查、测试扩展、证据整理和风险枚举；让人类保留方向、边界、裁决和责任。

## 旧模式的问题

| 旧模式 | 在 AI 时代的后果 |
| --- | --- |
| 人写模糊需求，AI 大量生成代码 | 代码差异过大，人工验收跟不上 |
| 同一个 AI 同时写实现和测试 | 测试会跟着错误实现走，容易自证正确 |
| 人类先做完整代码审查 | 人成为产能瓶颈，也容易疲劳漏审 |
| 只看 CI 是否通过 | 不能证明需求正确、测试有效、安全无回归 |
| 需求与代码弱连接 | 功能漂移、漏实现、顺手重构都难发现 |

## 新模式的目标

```text
需求变清晰
上下文变可控
实现变小切片
审查先由 AI 并行完成
测试由 AI 反向质疑
证据由 AI 汇总
人类只裁决剩余风险
```

## AI Coding 的边界

AI Coding 不是完全自动化交付。它要求更强的事实源、更小的变更、更明确的验收标准和更严格的审查证据。没有这些约束，AI 只会把混乱加速。

适合交给 AI 规模化处理的工作：

- 读取代码库并定位相关文件。
- 按既有模式实现明确需求。
- 生成测试、补边界用例、运行验证命令。
- 审查代码差异，寻找缺陷、安全风险和契约遗漏。
- 生成验收证据包、风险摘要和修复计划。

必须由人类保留的工作：

- 判断需求是否值得做。
- 接受或拒绝高风险变更。
- 决定架构方向和事实源迁移。
- 判断用户体验、品牌语气和商业优先级。
- 对最终合并、发布和事故负责。
