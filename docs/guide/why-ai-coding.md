# 为什么需要 AI Coding 体系

AI 已经把“写代码”的成本打下来了，但软件交付不会因此自动变简单。真正的瓶颈会转到需求澄清、上下文组织、审查、验收、发布和治理上。

如果团队只是把 AI 当成补全工具，短期会觉得产出变快；过一段时间就会发现，代码差异变大了，测试更难信了，审查排队更长了，返工和回滚也更多了。

AI Coding 体系要解决的不是“让 AI 替代工程师”，而是重新分工：AI 负责规模化执行、审查、补测试、整理证据和枚举风险；人负责方向、边界、取舍、裁决和最终责任。

## 旧模式的问题

| 旧模式 | 到了 AI 时代会放大的问题 |
| --- | --- |
| 需求模糊，AI 直接开写 | 代码差异很快变大，人工验收跟不上 |
| 同一个 AI 写实现又写测试 | 测试容易围着错误实现自证正确 |
| 人先做完整代码审查 | 人变成第一瓶颈，疲劳后也容易漏看 |
| 只看 CI 是否通过 | CI 通过不等于需求正确、测试可信、安全无回归 |
| 需求、代码、测试弱关联 | 漏实现、功能漂移、顺手重构都很难发现 |

## 新模式的目标

```text
需求更清楚
上下文更可控
实现更小切片
AI 先并行审查
测试由另一个 AI 反向质疑
证据由 AI 汇总
人只裁决剩余风险
```

## AI Coding 的边界

AI Coding 不是完全自动化交付。恰恰相反，它要求更明确的事实源、更小的变更、更可验证的验收标准，以及更严格的审查证据。没有这些约束，AI 只是在加速混乱。

适合交给 AI 规模化处理的工作：

- 读取代码库并定位相关文件。
- 按既有模式实现清楚的小需求。
- 生成测试、补边界用例、运行验证命令。
- 审查代码差异，寻找缺陷、安全风险和契约遗漏。
- 生成验收证据包、风险摘要和修复计划。

必须由人保留的工作：

- 判断需求是否值得做。
- 接受或拒绝高风险变更。
- 决定架构方向和事实源迁移。
- 判断用户体验、品牌语气和商业优先级。
- 对最终合并、发布和事故负责。
