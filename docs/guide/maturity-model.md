# AI Coding 成熟度模型

成熟度模型用来判断团队现在在哪一层，以及下一步最该补什么。不要一上来就追求“全自动合并”。更稳的路线是：先把小切片、验证证据和 AI 审查做好，再谈更高程度的自动化。

| 等级 | 状态 | 典型特征 | 下一步 |
| --- | --- | --- | --- |
| L0 | 零散使用 | 只用聊天或补全生成代码 | 建立仓库规则和验证命令 |
| L1 | AI 辅助 | AI 写局部代码，人完整审查 | 强制小切片，建立需求到证据的映射 |
| L2 | Agent 实现 | AI 能读仓库、改代码、跑测试 | 增加独立、只读的审查 Agent |
| L3 | AI 先审 | AI 先审，人只看高风险摘要 | 建立审查 Agent 组和证据包 |
| L4 | 证据驱动交付 | 每个变更都有证据、风险和回滚说明 | 接入发布观测和反馈学习 |
| L5 | 受治理的自动化 | 低风险高度自动化，高风险强治理 | 建立度量、审计和安全红队 |

## 不建议跨级跳跃

最危险的跳跃，是从 L1 直接进入“AI 自动改、自动测、自动合并”。这会绕过事实源、审查和责任边界。短期看很快，长期会带来隐性技术债、测试污染和安全风险。

## 升级优先级

1. 把任务拆小，限制代码差异规模。
2. 让 AI 输出“需求 -> 代码 -> 测试 -> 证据”的映射。
3. 引入只读 AI 审查 Agent。
4. 引入测试质疑 Agent，专门审查测试是否可信。
5. 引入安全红队 Agent，专门审查高风险面。
6. 引入汇总裁决 Agent，汇总剩余风险和人工裁决项。
7. 把结果接入 CI、PR 评论和发布流程。
